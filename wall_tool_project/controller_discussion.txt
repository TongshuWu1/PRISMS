Purpose
-------
This file is a handoff note for another Codex or researcher working on the
wall-tool controller project. It records the current controller design,
important tuning decisions, lessons from failed ideas, run commands, and the
main research direction.

The intended application is a wall-cleaning or facade-inspection tool for a
large building/skyscraper. The system is a cable-supported drone/tool assembly:
the reel controls cable payout velocity, while the drone modules control force
and acceleration through their thrust axes. The main research interest is this
mixed-input control problem:

1. Reel/spool is best treated as a radial velocity/tension actuator.
2. Drone modules are best treated as tangential acceleration/force actuators.
3. The controller should use the cable efficiently, not ask the drones to do
   everything.
4. Smooth, fast, efficient motion matters more than forcing zero tilt.

Current chosen controller
-------------------------
The current controller is MIESC: Mixed-Input Energy-Shaping Controller.

High-level split:
  - Reel velocity regulates radial cable length/tension geometry.
  - Drone acceleration regulates tangential tracking and swing energy.
  - A reference governor slows the path when tracking, cable geometry,
    contact, thrust, or allocation risk is high.
  - A thrust allocator maps desired tangential force, radial assist force,
    and desired torque into the left/right drone thrusts.

MIESC details
-------------
The MIESC controller lives in cable_hybrid_controller/miesc.py and is called
from WallToolSimulator.step().

Radial part:
  radial_error = target_length - measured_line_length
  radial_velocity_error = target_length_dot - measured_line_velocity

  radial_accel_cmd =
      reference_radial_accel
      + omega_r^2 * radial_error
      + 2*zeta_r*omega_r * radial_velocity_error

  The result is clipped by max_radial_accel.

Tangential part:
  theta_error = target_theta - measured_theta
  theta_dot_error = target_theta_dot - measured_theta_dot
  tangential_error = line_length * theta_error
  tangential_velocity_error =
      line_length * theta_dot_error + line_velocity * theta_error

  nominal_tangential_accel =
      reference_tangential_accel
      + omega_t^2 * tangential_error
      + 2*zeta_t*omega_t * tangential_velocity_error

Swing energy:
  The controller forms a storage/swing-energy function based on tangential
  position and velocity error. It then projects/clips the tangential
  acceleration so the command tends to dissipate swing energy instead of
  adding energy during bad phases.

This is one of the publishable ideas: the controller is not a generic PID.
It uses the physical input split:
  - radial/reel velocity for cable length and cable efficiency,
  - tangential/drone acceleration for path tracking and anti-swing.

Reference trajectory
--------------------
The facade mission uses "coverage-smooth":
  simulator.set_corner_smooth_path(targets, corner_speed=COVERAGE_CORNER_SPEED)

The path is a serpentine cleaning/inspection path over the facade. Corner
velocities are generated so the tool does not stop at every lane end. Boundary
logic prevents smoothing from pushing the path outside the work area.

Important lesson: reducing the corner speed alone did not meaningfully fix the
observed turn oscillation or jerk. The visible turn behavior is more coupled to
body/cable dynamics and force allocation than to the path corner speed alone.

Reference governor
------------------
The governor is a reference-speed governor, not a force controller. It does
not directly command thrust or spool velocity. It decides how fast the
reference path should be fed to MIESC.

In each step:
  trajectory.advance(dt * reference_speed_scale)

Then the reference state is scaled:
  reference velocity     *= reference_speed_scale
  reference acceleration *= reference_speed_scale^2

So if reference_speed_scale = 0.5, the controller tracks an easier, slower
moving target.

There are two governor layers:

1. Main safety governor
   Location: WallToolSimulator._update_reference_speed_scale()

   It computes risk from:
     - thrust risk:
         if either drone nears max thrust
     - allocation residual risk:
         if desired force/torque cannot be produced well by the two modules
     - tracking error risk:
         if measured tool error grows beyond tracking_error_slowdown_m
     - cable geometry risk:
         if cable vertical efficiency is poor
     - contact risk:
         if contact force is too low/high or cleaning speed is too high

   It uses the worst risk:
     risk = max(thrust_risk, residual_risk, error_risk,
                contact_risk, 0.75 * geometry_risk)

   Then:
     target_scale = 1.0 - (1.0 - reference_speed_min) * risk

2. Contact-valid path governor
   Location: WallToolSimulator._contact_valid_reference_governor_scale()

   It further caps speed for facade work. It looks at:
     - proximity to turns
     - sharpness of the upcoming turn
     - whether the turn is near the facade boundary
     - cable geometry at the active target
     - tracking ratio
     - work-mode speed ratio

   The final target scale is:
     target_scale = min(main_safety_scale, contact_governor_scale)

The governor output is filtered:
  slowdown rate = reference_slowdown_rate = 2.5
  recovery rate = reference_recovery_rate = 1.1

So it slows down faster than it speeds back up.

Current chosen governor parameters:
  reference_speed_min = 0.15
  contact_governor_turn_distance_m = 1.20
  contact_governor_turn_min_scale = 0.18
  contact_governor_geometry_efficiency = 0.62
  contact_governor_geometry_min_scale = 0.28
  contact_governor_tracking_ratio = 0.42
  contact_governor_tracking_min_scale = 0.16
  contact_governor_speed_ratio = 0.68
  contact_governor_speed_min_scale = 0.30
  thrust_slowdown_fraction = 0.82
  residual_slowdown_fraction = 0.020

Important interpretation:
  The governor is the "do not ask impossible things" layer. If it is too
  aggressive, the robot looks slow or hesitant. If it is too weak, the tool can
  lose contact quality, saturate thrust, or excite oscillation.

Concern about the current governor
----------------------------------
The current governor is deterministic, but it is still heuristic. It uses
hand-designed risk formulas such as:
  - slow down when thrust fraction is high,
  - slow down when tracking error is high,
  - slow down near turns,
  - slow down when cable geometry is poor,
  - slow down when allocation residual is high.

This works as an engineering safety layer, but it is not the cleanest research
idea. The user specifically does not like that it feels heuristic. A better
publishable direction is to replace this hand-tuned risk governor with a
predictive reference governor or an MPC-style supervisor.

Recommended next controller architecture:

  State estimator, optional at first
        -> predictive reference governor / MPC supervisor
        -> MIESC low-level radial/tangential controller
        -> thrust allocator and reel velocity command
        -> plant/simulator

The low-level MIESC controller should remain the inner controller. The
predictive governor should sit above it and choose how fast the reference path
is allowed to move.

Predictive reference governor idea
----------------------------------
At every control step, instead of computing heuristic risk terms, the governor
should predict the future for a short horizon, for example 1.5 to 3.0 seconds.
It then chooses the fastest safe reference speed.

Simple deterministic version:
  1. Build candidate speed scales, for example:
       1.00, 0.90, 0.80, 0.70, ..., reference_speed_min
  2. For each candidate, simulate or roll out the plant/controller forward
     over the prediction horizon.
  3. Reject candidates that violate constraints.
  4. Among feasible candidates, choose the fastest one.
  5. Filter or rate-limit the chosen speed scale so it does not jump.

Possible constraints:
  - no cable slack
  - thrust below actuator limit
  - tension within safe range
  - tracking error below facade-work limit
  - contact force within valid cleaning range
  - body angular rate below inspection/cleaning limit
  - allocation residual below a small threshold
  - cable vertical efficiency above a useful threshold
  - payload speed below cleaning blur limit

Possible objective:
  maximize path progress
  minimize tracking error
  minimize drone power
  minimize spool/reel work
  minimize body rate
  minimize payload acceleration/jerk
  reward cable support fraction

This would make the "governor" much less heuristic: it becomes a deterministic
short-horizon constrained optimization or constrained search problem. It is a
strong research direction because it directly addresses the mixed-input nature
of the system: the reel and drones are not independent, so the supervisor must
preview whether a future speed request is feasible for both.

MPC supervisor idea
-------------------
A fuller version is MPC. Instead of only choosing one scalar reference speed,
the MPC could choose a short horizon of:
  - reference speed scale,
  - tangential acceleration modifier,
  - spool velocity/tension modifier,
  - possibly cable tension target.

The MPC objective could explicitly include:
  - fast coverage,
  - smooth turn reversals,
  - cable efficiency,
  - low drone power,
  - low allocation residual,
  - low body angular rate,
  - valid contact.

This is more powerful than a scalar predictive governor, but it is also harder
to implement and tune. Recommended progression:
  1. Keep MIESC as the inner controller.
  2. Replace heuristic governor with predictive scalar reference governor.
  3. If that works, extend to MPC-style multi-input supervision.

MPPI attempt and current decision
---------------------------------
An MPPI-supervised MIESC implementation was attempted. The research idea was:
  - keep MIESC as the inner dynamics-aware controller,
  - let MPPI act as a residual/predictive supervisor,
  - use GPU rollouts for performance,
  - visualize the chosen rollout horizon in the UI.

The implementation was reverted for now.

Why it was removed:
  - the GPU rollout backend was not cleanly implemented in this checkout,
  - early behavior was less smooth and less stable than the MIESC baseline,
  - the objective weights became over-complicated,
  - async supervisor timing made it hard to know whether MPPI was actually
    improving the motion,
  - the user did not like the extra reference/target abstraction that made the
    tool appear to chase a moving point instead of simply tracking the desired
    path.

Current decision:
  Do not continue from the old MPPI code. If predictive control is revisited,
  start from the clean MIESC baseline and design a simpler, more transparent
  predictive governor or MPC supervisor with clear objectives:
    hard constraints first,
    tracking error and smoothness primary,
    drone effort secondary,
    cable usefulness rewarded only when it reduces total effort,
    small regularization on reel motion, jerk, body rate, and residual action.

UI decision from the MPPI period:
  Keep the richer Qt evaluation UI, but remove MPPI-specific rollout/status
  panels. The current UI should show sensors, trajectory real-vs-desired, and
  left/right drone inputs.

Particle filter / estimator role
--------------------------------
A particle filter is primarily an estimator, not a controller. It should not be
used as the main replacement for the governor. It becomes useful when the
state is uncertain or partially hidden.

Particle filter, EKF, or UKF could estimate:
  - cable stretch and damping state,
  - unmodeled cable friction,
  - unknown wind force,
  - wall normal/contact state,
  - thrust effectiveness,
  - slowly varying payload/tool parameters.

Best future architecture with estimation:

  Particle filter / EKF / UKF estimates hidden state and uncertainty
        -> predictive governor or MPC chooses safe future motion
        -> MIESC tracks radial/tangential commands
        -> allocator commands drones and reel

Recommendation:
  Do predictive reference governor first. Add particle filtering later when
  disturbance, contact uncertainty, wind, or model mismatch becomes the main
  bottleneck. Do not start with black-box RL or particle filtering before the
  deterministic predictive controller is strong.

Cable tension and spool policy
------------------------------
The cable is modeled as a unilateral spring-damper with finite cable stiffness
and damping. The controller asks for efficient cable tension rather than
blindly maximizing tension.

The spool command is velocity control. In MIESC mode:
  WallToolSimulator._miesc_spool_velocity_policy()

The spool command includes:
  - line length error feedback
  - line velocity feedback
  - cable encoder error feedback
  - radial acceleration feedforward
  - tension feedback
  - rate limiting by miesc_spool_accel_limit_mps2

The goal is to keep the cable taut and useful while avoiding abrupt payout
changes and excessive spool work.

Drone force and allocation
--------------------------
After MIESC computes radial_accel_cmd and tangential_accel_cmd, the simulator
computes:
  desired_tangential_force
  desired_radial_drone_force
  desired_drone_force
  desired_attitude_torque

Then it solves a bounded allocation problem for left/right drone thrusts.
The drone modules have tilted axes, so force and torque authority are coupled.

Important practical point:
  If we command too much attitude torque, it steals authority from translational
  force tracking. This is why active force-based pre-tilt was rejected.

Attitude control
----------------
There are two attitude regimes:

1. Move mode
   Current chosen behavior:
     move_attitude_kp = 0.0
     move_attitude_kd = 1.40
     move_max_attitude_torque = 0.040
     attitude_target = nominal_attitude_rad

   Interpretation:
     During motion, we do not force the drone to zero tilt with a strong
     position servo. We mainly add passive damping to reduce oscillation.
     This preserves thrust authority for translation and avoids fighting the
     cable/drone geometry.

2. Hold mode
   Current behavior:
     hold_attitude_kp = 35.0
     hold_attitude_kd = 8.0
     hold_max_attitude_torque = 0.06

   Hold attitude target:
     _hold_equilibrium_attitude_for_position(position, tension_upper)

   The hold target is produced by:
     _hold_optimal_static_equilibrium(position, tension_upper)

   This static optimizer searches candidate body attitudes and cable tensions.
   For each candidate, it computes the cable direction, cable torque, drone
   thrust axes, and required left/right drone thrusts using the existing
   bounded allocation solver. It then chooses the lowest-cost feasible static
   hold solution.

   The hold cost is ordered conceptually as:
     1. small static wrench residual,
     2. low left/right drone thrust effort,
     3. useful but not excessive cable tension,
     4. weak regularization against unnecessary tilt.

   Current hold-search parameters in SimParams:
     hold_tension_search_steps = 32
     hold_attitude_search_steps = 12
     hold_efficiency_residual_weight = 80.0
     hold_efficiency_tilt_weight = 0.012

   Important:
     Zero tilt is not required. The stable hold pose is chosen for efficiency,
     not for visual uprightness. If holding on the side of the wall, the body
     is allowed to tilt toward a pose where the cable supports more load and
     the drones do less work.

   Example checked after the latest change:
     target = (0.90, 1.50)
     optimized hold target attitude approximately -29 deg
     settled actual attitude approximately -21 deg
     cable tension approximately 1.54 N
     left/right thrust approximately 0.39 N / 0.00 N

   Interpretation:
     The actual settled pose can differ from the static target because the
     cable spring, measured tension, allocator, and body dynamics settle into
     the nearest feasible dynamic equilibrium. The important behavior is that
     hold mode no longer drives the body back upright when uprightness is not
     useful.

Turn smoothness discussion
--------------------------
User observed that turns are not smooth when the drone needs to tilt the other
way. This is a real issue.

Tested idea:
  Active force-based move attitude target, where body tilt was commanded from
  the required drone force direction.

Result:
  Rejected. It made the body look lower-rate, but it caused the thrust
  allocation residual to become active most of the mission and coverage dropped
  badly. In one run coverage fell to about 0.357 because the governor slowed
  down almost the whole time.

Why it failed:
  The system is underactuated/coupled. With two tilted thrust modules, forcing
  attitude consumes the same actuator authority needed for tangential tracking.
  A simple attitude pre-tilt target can fight the allocation.

Accepted fix:
  Increase passive move attitude damping:
    move_attitude_kd: 0.8 -> 1.4

This reduced body-rate oscillation without hurting coverage.

What not to repeat
------------------
1. Do not force zero tilt during motion.
   It is not necessary and can make the system less stable or less efficient.

2. Do not add many ad-hoc damping/fix terms without a physical reason.
   The user explicitly wants a mathematically defined, research-quality
   controller, not a pile of patches.

3. Do not use active force-based attitude pre-tilt unless the allocation is
   redesigned at the same time.
   Simple pre-tilt caused large allocation residual and poor progress.

4. Do not assume corner speed alone fixes turn smoothness.
   Sweeps of corner speed had little effect on the main jerk/body coupling.

5. Do not focus on wind or cable bumping right now.
   The current tuning scenario intentionally has wind disabled. The present
   research focus is fast, smooth, efficient motion in nominal conditions.

Research direction / next good ideas
------------------------------------
The current controller is good enough to be a baseline, but not yet the final
publishable controller. Strong next directions:

0. Replace the heuristic governor with a predictive reference governor
   This is now the top recommended next step. The current governor is
   deterministic but heuristic. A short-horizon predictive reference governor
   would choose the fastest safe reference speed by rolling out future
   constraints instead of using hand-tuned risk formulas.

1. Allocation-aware attitude/force control
   Instead of commanding attitude separately, solve one small QP that chooses:
     left thrust, right thrust, cable tension/spool objective, and possibly
     attitude torque
   subject to actuator bounds and with costs on:
     tracking force error,
     torque/body-rate,
     cable support efficiency,
     thrust energy,
     allocation residual.

   This could make attitude smoothing real without stealing translational
   authority.

2. Turn-aware MPC / preview controller
   Use upcoming path curvature and cable geometry to smoothly plan:
     reference speed,
     tangential acceleration,
     spool velocity,
     cable tension target.

   This is likely better for the observed turn reversal problem than local
   damping alone.

3. Control barrier function governor
   Replace or formalize the current heuristic reference governor with CBF-like
   constraints for:
     thrust limit,
     contact validity,
     cable tautness,
     tracking error,
     work-region boundary,
     angular-rate limit.

4. Energy-shaping with explicit jerk/turn penalty
   The current MIESC handles swing energy but not directly payload jerk.
   A publishable extension could add a smoothness objective or acceleration
   preview while respecting the mixed reel/drone input split.

5. Learn a residual model, not the whole controller
   If learning is used, prefer model learning for residual dynamics:
     cable stretch/friction,
     thrust allocation residual,
     contact normal dynamics.
   Then use it inside MPC or the governor. Avoid black-box RL until the
   model-based baseline is strong.

Current conceptual summary
--------------------------
The controller should be described as:

  A mixed-input energy-shaping controller with a reference-speed safety
  governor for a cable-supported wall tool. Radial cable geometry is regulated
  through reel velocity and tension management, while tangential tracking and
  anti-swing behavior are regulated through drone acceleration. A governor
  slows the reference when actuator, allocation, cable geometry, contact, or
  tracking risk rises. During motion, attitude is damped but not forced to a
  zero-tilt target, preserving thrust authority for translation.

This is the current best controller direction.
